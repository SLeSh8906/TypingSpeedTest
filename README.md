# Typing Speed Test (C# Edition)

## Опис
Консольний застосунок на .NET 8 для перевірки швидкості друку.
Користувач має ввести запропонований текст на швидкість.

## Мета
Навчитися працювати з Git flow, гілками, SSH ключами та створити
базовий додаток на C# для роботи з рядками та часом.

## Контрольні питання та відповіді

### 1. Основи Git
* **Різниця між Git та GitHub:** Git — це консольна утиліта для контролю версій (працює локально), а GitHub — це веб-сервіс для хостингу репозиторіїв Git.
* **"Розподілена" система:** Кожен розробник має на своєму комп'ютері повну копію всієї історії проєкту, а не лише поточні файли.
* **git init vs git clone:** `git init` створює новий репозиторій у поточній папці, а `git clone` копіює існуючий репозиторій з віддаленого сервера.

### 2. Staging Area та Commits
* **Staging Area:** Це проміжна зона ("індекс"), де ми формуємо майбутній коміт. Дозволяє вибрати, які саме зміни увійдуть в історію.
* **git add . vs git add -A:** `git add .` додає всі зміни з поточної папки, а `git add -A` додає зміни з усього проєкту (включно з видаленнями, хоча в нових версіях Git вони працюють майже однаково).
* **Гарне commit message:** Має бути коротким, в наказовому способі ("Add login", а не "Added login") і відповідати на питання "Що робить цей код?".
* **Виправити останній коміт:** Використовувати команду `git commit --amend` (якщо зміни ще не відправлені на сервер).

### 3. Гілки та злиття
* **Гілка (Branch):** Це рухомий вказівник на коміт. Вони "легковажні", бо займають мінімум пам'яті (41 байт).
* **git merge vs git rebase:** `merge` з'єднує історії, створюючи новий "merge commit" (зберігає реальну хронологію). `rebase` переписує історію, накладаючи ваші зміни поверх іншої гілки (робить історію лінійною).
* **Fast-forward merge:** Відбувається, коли в основній гілці не було нових змін. Вказівник просто пересувається вперед без створення окремого коміту злиття.
* **Merge конфлікти:** Виникають, коли одні й ті самі рядки коду змінені по-різному в двох гілках. Щоб уникнути: частіше робіть `git pull` і коміти.
* **Rebase публічних комітів:** Заборонено, бо це змінює хеші комітів. У колег виникне розсинхронізація версій.

### 4. Віддалені репозиторії
* **Origin:** Стандартна назва (псевдонім) для основного віддаленого репозиторію.
* **git fetch vs git pull:** `fetch` завантажує зміни з сервера, але не змінює ваші файли. `pull` робить `fetch`, а потім одразу `merge`.
* **git push --force:** Переписує історію на сервері вашою версією. Це небезпечно, бо може видалити чужу роботу.
* **Видалити віддалену гілку:** `git push origin --delete назва_гілки`.

### 5. Скасування змін
* **git reset (--soft, --mixed, --hard):**
    * `--soft`: скасовує коміт, зміни залишаються в Staging.
    * `--mixed`: скасовує коміт і Staging, зміни залишаються у файлах.
    * `--hard`: повністю видаляє зміни, повертає стан до коміту.
* **git reset vs git revert:** `reset` відкочує час назад (змінює історію). `revert` створює *новий* коміт, який скасовує дії попереднього (безпечно для історії).
* **Скасувати git add:** `git restore --staged файл`.
* **Повернути видалений файл:** `git restore файл` або `git checkout HEAD файл`.

### 6. Безпека
* **Що не можна комітити:** Паролі, API ключі, папку `node_modules`, бінарні файли (`/bin`, `/obj`).
* **Якщо закомітили пароль:** Негайно змінити пароль у сервісі. Видалити його з історії Git за допомогою інструментів типу BFG Repo-Cleaner.
* **HTTPS vs SSH:** SSH краще, бо використовує ключі шифрування і не вимагає вводити пароль щоразу.
* **Personal Access Token:** Це спеціальний код замість пароля для доступу через HTTPS (вимога безпеки GitHub).

### 7. GitHub
* **Pull Request (PR):** Запит на злиття змін з однієї гілки в іншу з можливістю обговорення коду (Code Review).
* **Issues vs PR:** Issues — це опис задачі або багу. PR — це код, що вирішує цю задачу.
* **Fork:** Копія чужого репозиторію у ваш акаунт. Потрібен, щоб пропонувати зміни в чужі проєкти.
* **GitHub Actions:** Інструмент для автоматизації (CI/CD) — автоматичний запуск тестів чи деплою при пуші.

### 8. Best Practices
* **Частота комітів:** Якомога частіше. Кожен коміт — одна логічна дія.
* **Життя feature branch:** Гілка має існувати недовго (дні), поки виконується задача.
* **Git workflow:** Популярні підходи: Git Flow, GitHub Flow, Trunk Based Development.
* **Комітити залежності:** Ні. Всі бібліотеки мають завантажуватися автоматично (через `restore` / `npm install`).
